# PORTFOLIO_AUTOMATION_RULES.md

## Product model

This product is a fully automated artist application system.

The user must only review two things:

1. Opportunity selection review
   - The system researches many possible exhibitions, residencies, grants, gallery open calls, and similar opportunities.
   - The system compares them against the artist database, artwork database, CV, prior applications, language ability, location, medium, themes, and eligibility.
   - The system recommends a small shortlist, usually around 5 opportunities.
   - The user chooses which opportunities to apply for.

2. Final submission review
   - After the user selects an opportunity, the system automatically creates the final application package.
   - The user only approves submission or requests a revision.
   - The user must not be asked to manually choose images, approve captions one by one, select layouts, select colors, repair portfolio pages, or inspect every page.

The system must behave like an automatic application assistant and portfolio designer, not a manual portfolio editor.

---

## Non-negotiable automation rule

Do not add extra review steps.

Forbidden review steps:
- asking the user to choose portfolio images
- asking the user to select layout templates
- asking the user to pick colors
- asking the user to approve every portfolio page
- asking the user to write captions manually
- asking the user to manually fix ugly pages
- asking the user to decide page rhythm
- asking the user to manually compare open calls one by one

Allowed user reviews:
1. choose opportunities from a shortlist
2. approve or reject final submission package

Only ask the user for input when there is a true blocker:
- missing required material
- unclear eligibility
- required fee/payment
- login/captcha
- legal declaration
- privacy risk
- irreversible submission action
- missing usable artwork files
- required physical artwork availability is unclear

Ordinary quality issues must be fixed automatically.

## Portfolio output goal

The portfolio PDF is the primary external-facing artifact.

When no opportunity-specific page limit exists:
- target: 20 pages
- preferred range: 18-22 pages
- acceptable range: 16-24 pages

If an opportunity explicitly requires a different page count, follow the opportunity requirement.

The final portfolio must look like a mature artist application PDF, not:
- a web preview
- a test export
- a generic SaaS report
- a plain image dump
- a white webpage printed to PDF
- a commercial brochure
- a draft packet

External-facing portfolio files must never contain:
- TEST
- PREVIEW
- Web preview
- browser preview
- mock
- draft
- placeholder
- unknown
- N/A
- TBD
- generated by AI
- ready to copy
- selected for this open call
- internal rationale

If metadata is missing, omit it externally and record it internally.

## Portfolio structure requirements

Default portfolio rhythm should include, when material allows:

1. Cover
2. Short artist/practice statement
3. Project opener
4. Installation or overview page
5. Primary work page
6. Detail/material/context page
7. Second project opener
8. Series overview grid
9. Primary work page
10. Two-image spread or detail page
11. Third project opener
12. Image research/context grid
13. Primary work page
14. Detail/process/context page
15. Fourth project or transition page if available
16. Primary image or installation documentation
17. Detail/context page
18. Selected works list
19. Short CV/exhibitions/residencies summary
20. Contact page

This is a model, not a rigid template. Adapt to material type.

The portfolio must use at least 5 meaningful layout strategies when enough material exists.

Allowed layout strategy examples:
- cover
- short_statement
- project_opener
- series_overview_grid
- single_work_full_page
- single_work_with_detail
- two_image_spread
- installation_overview
- installation_with_details
- image_research_grid
- text_image_context
- detail_page
- process_context
- selected_works_list
- contact_or_cv_summary

No more than 3 consecutive pages may use the same layout strategy.

Do not simply duplicate the same image/page format to hit the page target.

## Visual quality requirements

The portfolio must pass an automatic aesthetic gate before it can be marked ready.

The aesthetic gate must check:
- page count target/range
- PDF exists
- PDF is primary artifact
- no forbidden external-facing terms
- layout strategy diversity
- repeated layout runs
- excessive white-only pages
- weak cover
- excessive blank space
- images too small for page role
- distorted images
- missing images
- captions too long or too dense
- all pages having the same visual structure
- missing project rhythm
- weak image/text hierarchy
- whether output feels like a formal PDF rather than an HTML preview

Minimum readiness:
- portfolio.pdf exists unless the opportunity does not require PDF
- page count meets opportunity/default range
- no missing/unreadable images
- no forbidden words
- at least 5 layout strategies when enough material exists
- no more than 3 consecutive same-layout pages
- aesthetic score >= 85
- professional PDF score >= 85

If the gate fails, auto-repair and rerender up to 3 rounds.

If still failing, mark the package as quality_blocked.
Do not mark it ready.

---

## Auto-repair requirements

The system must automatically repair ordinary portfolio problems.

Auto-fix:
- too few pages
- too many pages
- repeated layouts
- weak cover
- too much white-background monotony
- pages that look like web previews
- overlong captions
- missing section rhythm
- too many tiny images
- too many single-image pages
- weak typography hierarchy
- duplicate image usage
- generic AI-like statement text
- external-facing draft/test/preview language

Only block if:
- no usable artwork files exist
- image files are missing or unreadable
- required opportunity material is unavailable
- opportunity requirements conflict
- file-size/page-count constraints are impossible
- eligibility/fees/submission method are unclear

---

## Image selection rules

The user should not manually choose images.

The system should select works by:
- opportunity fit
- artist themes
- project diversity
- recency
- medium fit
- image quality
- availability
- prior application history
- need for coherent project groups

The portfolio should usually include multiple project groups when available.

Avoid selecting only one series unless the opportunity or artist database clearly supports a single-project portfolio.

If only one project is genuinely available, make that limitation internal and design the PDF as a focused project portfolio instead of pretending it is broad selected works.
